## S1 — Cross-session recall
Wrote `dogfood_canary.md` with `topics: [testing]` and a sentinel string in the body, then:

```bash
echo '{"prompt":"run a vitest spec quickly to mock the network call"}' \
  | MEMORY_ROUTER_DIR=$MEM node dist/hooks/user-prompt-submit.js
# → hookSpecificOutput.additionalContext lists the canary at "topic · 1.00"
```

**Result:** ✅ pass.

**Caveat:** the first attempt used `topics: [dogfood-canary-xyz123]` (a custom topic
not in `TOPIC_PATTERNS`) and produced empty output. Memory-router has a closed
set of 5 hardcoded topics (`deployment`, `destructive_ops`, `workflow`,
`security`, `testing`) baked into `topic-patterns.ts` as `Record<Topic, RegExp[]>`,
so unknown topics are a type-level contract violation that the runtime gate
silently treats as a no-match. The follow-up `b0457c1d` is therefore scoped as
a **schema/lint check** at memory-write time, not a runtime warner.

## S3 — Conflict
Wrote two `feedback` memories with the same `topics: [workflow]` and opposing
imperatives ("ALWAYS amend" vs "NEVER amend"), then:

```bash
echo '{"prompt":"creating a PR and addressing review findings on the branch"}' | …
# both memories injected, score 1.00 each, no dedup, no warning
```

**Result:** ⚠️ behavior documented. `dedupeAndRank` (`router.ts`) only collapses
the same memory matched by multiple gates (keyed on `memory.id`); two
**distinct** memories matching the same topic both pass. Router faithfully
forwards both — the model is left to reconcile at runtime. Confirms the
existing follow-up `9a7737a7` (Memory-Konflikt-Detektor) is real and worth
shipping.

## S4 — MEMORY.md truncation
Out of scope for this hook. `loadMemoriesFromDir` skips entries where
`entry === 'MEMORY.md'`, so MEMORY.md content never reaches the gates. The
200-line truncation is a Claude-Code-harness contract on the auto-memory
CLAUDE.md context loader, not on memory-router.

## S5 — Frontmatter tolerance
Two test files, two different code paths:

- **`dogfood_no_topics.md`** — valid YAML with `name`+`type` set, just no
  `topics:` field. `parseMemoryFile` accepts it (the only required keys are
  `name` and `type`). It loads but never matches: the topic gate reads
  `memory.frontmatter.topics ?? []`, gets an empty list, and skips. So this
  memory is **loaded but invisible** rather than rejected.

- **`dogfood_broken_yaml.md`** — unbalanced bracket + invalid extra key. The
  YAML parser throws, `parseMemoryFile` catches and returns `null`, so the
  loader **drops** it entirely.

```bash
echo '{"prompt":"PR review on the workflow branch"}' | …
# exit 0, no crash, neither broken memory appears in output, well-formed memories still served
```

**Result:** ✅ pass — fail-open behavior is correct for production.

**Follow-up:** silent skip is the right default but leaves memory authors in
the dark about dead-weight files (the no-topics case is especially silent —
the file *looks* loaded). Tracked as `6d8dd01b` (opt-in stderr warning on
rejected/never-matched frontmatter).
